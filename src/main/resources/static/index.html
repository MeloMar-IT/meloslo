MeloSlo Documentation

What is OpenSLO?

OpenSLO is an open-source specification for defining Service Level Objectives (SLOs) in a common, vendor-neutral format. It allows teams to define their reliability goals, error budgets, and indicators in a consistent way across different monitoring tools.

Core Concepts:

  • Service: A logical collection of SLOs.
  • SLI (Indicator): A metric that tells you how your service is performing (e.g., Latency, Error Rate). Note: All metrics must be in the range 0 to 100.
  • SLO (Objective): A target value for an SLI over a specific time window. Multiple SLIs can be linked to a single SLO.
  • Data Source: A record defining where metrics come from (e.g. Prometheus). Supports custom Refresh Rates (min 15m).
  • Error Budget: The amount of unreliability you can afford before violating your SLO.
  • Alerting Source: A record defining a webhook (e.g. Slack) to notify when an SLO is breached. Includes variables like ${SLO_NAME} and 6h anti-spam.
  • Background Processing: All heavy tasks (fetching metrics, sending alerts) run in background threads with a 5-minute timeout watchdog for stability.
  • Status Thresholds: SLOs are Healthy (Budget ≥ 25%), Warning (0% to 25%), or Breaching (< 0%).

Authentication and Access Control

MeloSlo implements secure authentication and department-based access control.

Role: Administrator (Full Access)

Users without an assigned department can see and manage all records across the entire application and access the User Admin and Database Query tools.

Role: Restricted User

Users can be assigned to one or more departments (separated by commas). They only see records belonging to any of those departments (e.g., 'Finance,Engineering').

How to use MeloSlo

1. Create a Service

Start by defining the Service that owns the reliability goals. Go to 'Services' and click 'Create'.

2. Define Indicators (SLIs)

Define how you measure performance. You can specify Prometheus queries or other metric sources in the YAML specification. Important: All metrics should be normalized between 0 and 100.

3. Set Objectives (SLOs)

Link your Service and SLIs together and set target percentages. You can link multiple SLIs to a single SLO to monitor multiple performance indicators.

4. Manage Metadata

Each record can be tagged with a Department and a Manager to ensure clear ownership and accountability.

5. Generate Reports

Go to the 'Reports' view to generate professional annual PDF performance reports for your Services and SLOs. Gaps in data are automatically handled.

6. Trend Prediction

MeloSlo includes an advanced trend prediction engine that uses additive modeling (linear trend + seasonality) to forecast future error budget consumption based on historical data. Look for the 'Trend Analysis' section in SLO details.

7. Configure Alerting

Create an 'Alerting Source' with your webhook URL and JSON payload. Then link it to your SLOs to get notified when they are breaching. Use ${SLO_NAME}, ${SLO_STATUS}, and ${SLO_VALUE} variables.

8. Database Maintenance

Administrators can use the 'Query Tool' to run SELECT queries against the internal database to audit records or inspect table structures.

9. Thread Stability

MeloSlo uses a background thread pool for all non-blocking operations. A built-in watchdog ensures that no single task (like a slow metric fetch) consumes resources for more than 5 minutes.

Sample Specification

Standard SLO:

apiVersion: openslo/v1
kind: SLO
metadata:
  name: api-availability
spec:
  service: my-service
  description: 99.9% of requests must return 2xx or 3xx
  indicator:
    metricSource:
      type: prometheus
      spec:
        query: sum(rate(http_requests_total{status=~"2..|3.."}[5m])) / sum(rate(http_requests_total[5m]))
  objectives:
    - ratioMetrics:
        target: 0.999

SLI with Multiple Metric Sources:

apiVersion: openslo/v1
kind: SLI
metadata:
  name: multi-cloud-latency
spec:
  metricSources:
    - type: prometheus
      spec:
        query: histogram_quantile(0.95, sum(rate(http_duration_bucket{env="aws"}[5m])) by (le))
    - type: datadog
      spec:
        query: avg:trace.http.request.duration{env:azure}.p95
  description: Latency metrics from both AWS and Azure

DataSource Template (Prometheus):

apiVersion: openslo/v1
kind: DataSource
metadata:
  name: prometheus-datasource
spec:
  type: prometheus
  config:
    url: http://prometheus:9090
# Note: Refresh rate (min 15m) is managed via the UI and database.

Annual Performance Reports

MeloSlo allows you to generate professional PDF reports covering the performance of your Services and SLOs over the last 365 days.

  • Consolidated View: Select multiple Services or SLOs to include in a single report.
  • Annual Performance: The report automatically calculates the average performance and error budget for the last year.
  • Status Tracking: Includes health indicators (Healthy, Warning, Breaching) based on annual targets.
  • Service Aggregation: Selecting a Service automatically includes all linked SLOs in the report.

Database Schema

Table openslo_records

Stores all core entities: Services, SLIs, SLOs, DataSources, and AlertingSources. Each record is mapped to a Department and ITAL (IT Area Lead).

id: PK (Long)
kind: Record Type (Enum)
name: Unique System Name
display_name: Human Name
spec: Raw YAML/JSON Config
department: Assigned Group
manager: Manager
alert_url: Webhook URL
alert_payload: JSON Template
service_id: FK (Service Parent)
alerting_source_id: FK (AlertSource)

Table sli_metric

Time-series data for all SLIs. Metrics are collected by the scheduler based on the DataSource refresh rate.

id: PK (Long)
timestamp: Collection Time
metric_value: Measured Value (Range: 0-100)
sli_id: FK (Target SLI)
datasource_id: FK (Source)

Table melo_users

Manages application users and their access control permissions (Role-Based and Department-Based).

id: PK (Long)
username: Login Unique Name
email: User Email
departments: CSV list of Depts
password: BCrypt Encoded

Table slo_slis

Join table for the Many-to-Many relationship between SLOs and SLIs (Objectives and Indicators).

slo_id: FK (SLO)
sli_id: FK (SLI)
